# Диздок Millionaire Quiz Engine

## 1. Назначение продукта

`Millionaire Quiz Engine` — переиспользуемый браузерный движок викторин в
формате "Who Wants to Be a Millionaire?". Репозиторий содержит сам движок,
application shell со списком игр и набор готовых content-модов.

Цель проекта — быстро собирать тематические викторины с разным визуальным
стилем, кампаниями, вопросами, ассетами и аудио, не переписывая игровую
логику.

Публичный runtime:

- статический сайт на Vite/React;
- SPA с base path `/millionaire/`;
- GitHub Pages deployment через `npm run deploy`;
- content-agnostic engine, который получает игру через `GameConfig`.

## 2. Целевая аудитория

Основной пользователь — игрок, который хочет пройти короткую тематическую
викторину с напряжением классического quiz show.

Второй пользователь — автор игры. Для него важны:

- понятная структура `src/games/<gameId>/`;
- возможность добавлять кампании и вопросы без правки engine logic;
- управляемые темы, строки, ассеты и аудио;
- быстрый feedback через dev server, тесты и manifest generation.

## 3. Core loop

1. Игрок открывает hub.
2. Выбирает игру на экране game selector.
3. Подтверждает звук или отказывается от него.
4. Выбирает кампанию, если игра содержит несколько кампаний.
5. Отвечает на серию вопросов.
6. Использует `lifeline`, если нужна помощь.
7. После правильного ответа продвигается по `prizeLadder`.
8. При ошибке, победе или добровольном уходе попадает на end screen.
9. Может вернуться к старту или выбрать другую кампанию/игру.

## 4. Игровые состояния

Движок должен различать как минимум следующие стадии:

- loading shell;
- start screen;
- campaign selection;
- active question;
- answer feedback;
- lifeline result overlay;
- victory;
- defeat;
- retreat.

Чистая state machine находится в `src/engine/game/state/`. React-хуки в
`src/engine/ui/hooks/` связывают её с UI, audio и config.

## 5. Вопросы

Вопрос — атомарная единица прохождения.

Ожидаемые свойства вопроса:

- текст вопроса;
- варианты ответов;
- правильный ответ;
- сложность или принадлежность к пулу кампании;
- optional explanation или metadata, если конкретная игра это поддерживает.

Вопросы должны быть детерминированно пригодны для тестирования. Если
используется перемешивание, оно должно быть локализовано в engine utilities,
а не размазано по UI.

## 6. Кампании

Кампания — тематический набор вопросов внутри игры. Она может задавать:

- `campaignId`;
- отображаемое название;
- тему;
- пул вопросов;
- изображения для start/play/end состояний;
- campaign-specific music.

Кампании позволяют одной игре иметь несколько маршрутов прохождения:
например разные персонажи, уровни сложности, сюжетные арки или тематические
наборы.

## 7. `prizeLadder`

`prizeLadder` — каноническое имя лестницы наград.

Он задаёт:

- список значений наград;
- гарантированные уровни;
- текущую позицию игрока;
- отображение прогресса в `PrizeLadderPanel`;
- расчёт результата при retreat, victory и defeat.

Нельзя заменять термин `prizeLadder` на `money ladder`, `score ladder` или
другие локальные варианты в коде и документации.

## 8. `rewardKind`

`rewardKind` описывает тип финального результата:

- `trophy` — тематическая немонетарная победа;
- `money` — денежная награда в классическом стиле;
- `defeat` — проигрыш.

Игры могут визуально переосмыслять награды, но контракт результата должен
оставаться предсказуемым для engine UI.

## 9. `lifeline`

`lifeline` — специальная помощь игроку. Не использовать термин `hint` для
этого механизма.

Поддерживаемые виды:

- `fifty` — убрать часть неправильных ответов;
- `phone` — совет персонажа или условного помощника;
- `audience` — распределение мнения аудитории;
- `host` — помощь ведущего;
- `switch` — замена вопроса;
- `double` — право на одну ошибку.

Каждый `lifeline` имеет:

- enabled/disabled состояние в `GameConfig`;
- отображаемое имя;
- иконку;
- состояние доступности в игровой сессии;
- результат, который показывается в overlay/panel.

`lifeline` должен быть одноразовым, если конкретная механика явно не
предусматривает другое поведение.

## 10. UX и экраны

### Game selector

Показывает доступные игры и метаданные из `manifest.ts`. Карточка игры должна
оставаться лёгкой: selector не должен тянуть тяжёлый `GameConfig` всех игр
ради первого экрана.

### Loading

Loading UI оркестрирует фазы загрузки engine, config и ассетов. Отсутствие
опционального ассета не должно ломать запуск игры.

### Sound consent

Браузеры ограничивают autoplay. Первый вход в игру должен получить user
gesture и позволить игроку включить или выключить звук.

### Start screen

Показывает тему игры, campaign entry points и стартовое настроение. Если
кампания одна, UI может вести игрока напрямую, но поведение должно оставаться
предсказуемым.

### Game screen

Основной экран содержит:

- вопрос;
- ответы;
- `lifeline` controls;
- `PrizeLadderPanel`;
- header/portal visual;
- audio feedback.

### End screen

End screen показывает итог, награду, defeat/victory/retreat состояние и
дальнейшие действия.

## 11. Визуальный стиль

Каждая игра может иметь собственную тему, но engine UI должен сохранять
узнаваемую структуру quiz show:

- фокус на вопросе и ответах;
- ясные интерактивные состояния;
- читаемый контраст;
- стабильная layout-сетка;
- аккуратная адаптивность;
- отсутствие зависимости от одного обязательного фонового изображения.

CSS engine-уровня scoped под `.engine`, чтобы стили не протекали в app shell.

## 12. Аудио

Аудио делится на:

- music;
- SFX;
- voice lines;
- generated или silent fallback.

Принципы:

- звук не стартует без согласия игрока;
- отсутствие файла не ломает игру;
- SFX должны иметь низкую задержку;
- voice files не должны агрессивно прелоадиться, если это раздувает память;
- end/victory/defeat music выбирается из game/campaign context.

## 13. Ассеты и манифесты

Runtime-ассеты лежат в `public/`. Два основных источника:

- `public/asset-manifest.json` для `AssetLoader`;
- `public/**/manifest.json` для image slideshows/header.

Preload levels:

- `level0` — game selector assets;
- `level1` — выбранная игра и start screen;
- `level1_1` — выбранная кампания;
- `level2` — gameplay и end assets.

Подробное фактическое поведение описано в
`docs/asset-loading-strategy.md`.

## 14. Контентная модель игры

Игра должна описываться через `GameConfig` и связанные файлы:

```text
src/games/<gameId>/
  manifest.ts
  index.ts
  config.ts
  strings.ts
  icons.tsx
  campaigns/
    <campaignId>/
      questions.ts
      theme.ts
```

`GameConfig` является главным контрактом между content-модом и engine.
Engine не должен импортировать конкретную игру напрямую ради доступа к
вопросам или ассетам.

## 15. Текущие игры

- `poc` — минимальная демонстрация возможностей engine.
- `axis` — тематическая игра с отдельной стилистикой и кампаниями.
- `bg3` — игра по Baldur's Gate III.
- `nnr` — игра по нейросетям/курсу NNR.
- `sky-cotl` — игра по Sky: Children of the Light.
- `transformers` — игра по Transformers comics.

Список в README должен обновляться вместе с добавлением или удалением игр.

## 16. Критерии качества

Фича считается готовой, если:

- поведение покрыто релевантными тестами;
- `npm test` проходит;
- `npm run lint` проходит для кодовых изменений;
- `npm run build` проходит для изменений сборки, ассетов или public surface;
- UI-поток проверен в браузере для визуальных изменений;
- документация в `docs/` не противоречит коду;
- игра продолжает запускаться при отсутствующих optional manifests/assets.

## 17. Out of scope

На текущем этапе диздок не утверждает:

- редактор игр;
- серверное хранение прогресса;
- multiplayer;
- авторизацию;
- monetization;
- загрузку пользовательских паков из внешнего источника;
- юридическую модель лицензирования сторонних ассетов.

Если один из этих пунктов станет задачей, нужно создать отдельный план в
`docs/superpowers/plans/` и обновить этот диздок.
