# Агентский workflow

Этот документ описывает локальный рабочий контекст агентов и правила
документационных handoff-ов. `.agent/` — локальная память сессии, а не
каноническая документация проекта.

## Локальный контекст

Каждый агент работает в собственной папке:

```text
.agent/
  agents/
    <agent-id>/
      current-task.md
      worklog.md
```

Рекомендуемый `agent-id`: `<tool>-<YYYYMMDD-HHMMSS>-<short-slug>`.

`current-task.md` — краткий живой статус задачи. Минимальная структура:

```markdown
# Текущая задача

## Цель

## Статус

## План

## Touched files

## Риски и допущения
```

`worklog.md` — журнал фактических действий. Туда пишутся только полезные
события: прочитанные документы, важные выводы, принятые решения, выполненные
проверки и обнаруженные ограничения.

## Правила работы

- `.agent/` не коммитится.
- Каждый агент редактирует только свой каталог внутри `.agent/agents/`.
- Перед правкой нового файла агент сверяется с чужими `Touched files`.
- Если зоны ответственности пересекаются, сначала зафиксируй handoff или
  разведи `Touched files`.
- Если работа передаётся другому агенту, handoff фиксируется в обоих
  `worklog.md`.

## Документационные изменения

Документация обновляется вместе с изменением поведения. Для выбора нужного
файла используй `docs/workflows.md`.

При создании нового канонического документа:

1. Добавь файл в `docs/repository-layout.md`.
2. Добавь ссылку в `README.md`, если документ нужен новым разработчикам.
3. Обнови `AGENTS.md`, если документ становится обязательным перед началом
   каждой задачи.

## Крупные архитектурные задачи

Крупные задачи оформляются как исполняемые планы в
`docs/superpowers/plans/`.

План должен содержать:

- цель и границы работы;
- текущую архитектуру;
- целевую архитектуру;
- решения по терминологии;
- список задач с чекбоксами;
- ожидаемые файлы;
- проверки после каждого крупного шага;
- явные out-of-scope пункты.

Во время выполнения план обновляется по факту. Нельзя отмечать шаг
завершённым до проверки результата.
