# AGENTS.md

Эти правила обязательны для каждого ИИ-агента и каждой новой рабочей сессии в
этом репозитории.

## Перед началом задачи

1. Прочитай `README.md`, `docs/workflows.md`, `docs/repository-layout.md`,
   `docs/game-design.md`, `docs/styleguide.md` и
   `docs/asset-loading-strategy.md`.
2. Посмотри существующие агентские контексты в
   `.agent/agents/*/current-task.md`, если каталог есть.
3. Выбери уникальный `agent-id`. Рекомендуемый формат:
   `<tool>-<YYYYMMDD-HHMMSS>-<short-slug>`.
4. До изменения кода или продуктовой документации создай локальный контекст:
   `.agent/agents/<agent-id>/current-task.md`.
5. В `current-task.md` зафиксируй:
   - цель задачи;
   - текущий статус;
   - план действий;
   - `Touched files`;
   - риски и допущения.
6. Создай или обнови `.agent/agents/<agent-id>/worklog.md` и веди в нём
   краткий журнал действий, выводов и проверок.

`.agent/` — локальный рабочий контекст. Его нельзя коммитить и нельзя
использовать вместо канонической документации в `docs/`.

## Во время работы

- Соблюдай правила из `docs/`.
- Работай только со своей папкой `.agent/agents/<agent-id>/`.
- Не переписывай чужие агентские контексты. Если зоны ответственности
  пересекаются, сначала зафиксируй handoff или разведи `Touched files`.
- Если меняется поведение движка, приложения, ассетов, контента, сборки,
  маршрутизации, публичных типов или рабочих процессов, обнови
  соответствующий документ в `docs/` в той же задаче.
- Если подходящего документа нет, создай его и добавь ссылку в `README.md`
  или `docs/repository-layout.md`, если документ становится каноническим.
- Держи `docs/repository-layout.md` актуальным при изменении структуры
  каталогов, назначения файлов, скриптов или генерируемых артефактов.
- Не создавай новые слои, каталоги и абстракции без явной пользы для текущей
  задачи.
- Не храни реальные закрытые ассеты или данные в репозитории. Игровые ассеты
  сторонних правообладателей должны оставаться только там, где проект уже
  ожидает runtime-ассеты, и не должны попадать в тестовые фикстуры как
  обязательная зависимость.
- Во всех новых путях внутри репозитория используй латиницу, цифры, дефисы и
  подчёркивания. Кириллица допустима в содержимом документов и отображаемых
  игровых текстах.

## Код и архитектура

- `src/engine` — переиспользуемый quiz engine. Он не должен зависеть от
  конкретных игр, маршрутов приложения или DOM вне своего UI-слоя.
- `src/app` — shell приложения и каталог игр.
- `src/pages` — route-level страницы.
- `src/games` — контент и конфигурации конкретных игр.
- `public` — runtime-ассеты и сгенерированные манифесты.
- Следуй терминологии проекта: `lifeline`, `prizeLadder`, `rewardKind`,
  `GameConfig`, `campaign`, `gameId`.
- Импорты между верхними слоями делай через aliases из `tsconfig.json` и
  `vite.config.ts`.
- Если задача затрагивает текущий план разделения `engine`, `game` и `hub`,
  учитывай документ в `docs/superpowers/plans/`, но не считай его уже
  реализованной архитектурой.

## Проверки

- Для любых изменений кода запускай релевантные проверки. Базовый минимум:
  `npm run typecheck`, `npm run lint`, `npm test`, при изменениях сборки или
  ассетов — `npm run build`.
- В репозитории используются versioned git hooks из `.githooks/`. Если hooks
  ещё не подключены в текущем clone, запусти `npm run hooks:install`.
- Перед коммитом ориентируйся на `npm run quality:commit`; перед push — на
  `npm run quality:push`.
- Не обходи hooks через `--no-verify`, кроме случая, когда пользователь явно
  попросил аварийный обход и принял риск.
- Для изменений UI, маршрутов, пользовательских сценариев, стилей или
  browser-side поведения нужна проверка в браузере. Минимум: открыть
  затронутый сценарий, пройти основной flow и просмотреть результат.
- Для документальных правок без изменения кода достаточно проверить связность
  ссылок, соответствие текущей структуре репозитория и отсутствие
  противоречий с `README.md`.
- Если проверку невозможно выполнить, явно запиши причину в `worklog.md` и в
  финальном ответе.

## После завершения

- Обнови свой `current-task.md`: фактический статус, итоговые `Touched files`,
  выполненные проверки и остаточные риски.
- Допиши в `worklog.md`, что сделано и чем подтверждено.
- Не оставляй устаревшие планы в своей `.agent/agents/<agent-id>/` папке.
- В финальном ответе кратко перечисли изменённые канонические документы и
  проверки.
